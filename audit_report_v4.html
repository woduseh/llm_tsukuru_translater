Tsukuru Extractor v3.0.0 — 프로덕션 준비도 게이트 리뷰 | 2026-03-03

A. 개요 및 기술 스택

RPG Maker MV/MZ 및 Wolf RPG Editor 게임의 텍스트를 추출, 번역(Gemini LLM), 적용하는 Windows 전용 Electron 데스크톱 앱이다. Vue 3 SPA를 Vite로 빌드하며, Main 프로세스의 IPC 핸들러를 통해 파일 시스템 작업과 LLM API 호출을 처리한다. JSON 구조 검증(줄밀림 방지) 및 자동 수정 기능이 핵심 차별점이다.

C. 주요 발견 사항

1. 지속 가능성

비의도적 결함
이중 파일 규칙(.ts + .js)으로 5,260 LOC 중복

Electron Main 프로세스가 컴파일된 JS를 직접 실행하는 구조 때문에 모든 .ts에 .js 동반 파일을 커밋해야 한다. 빌드 스텝 없이 Electron이 바로 실행하는 설계 선택의 결과이나, 수동 동기화는 오류 유발 요인이다.

src/ 전체 — 44개 .js 파일
비의도적 결함
globalThis에 16개 전역 변수 의존

settings, gb, mwindow, WolfCache 등의 상태를 globalThis에 직접 할당. 모듈 간 암묵적 결합을 만들며, 단위 테스트에서 모킹이 어렵다.

globals.d.ts, src/ipc/shared.ts, src/js/rpgmv/extract/formatter.ts
강점
IPC 핸들러 6개 모듈 분리 + Copilot 인스트럭션 문서 아키텍처 기록

main.ts → 6개 IPC 모듈로 분리가 잘 되어 있고, .github/copilot-instructions.md가 사실상 아키텍처 문서 역할을 수행.

2. 엔지니어링 재현성

강점
CI/CD 파이프라인 + 155개 테스트 전체 통과

GitHub Actions에서 tsc · ESLint · vitest coverage · electron-builder 검증까지 자동화. package-lock.json 커밋으로 재현성 확보.

.github/workflows/ci.yml
비의도적 결함
모듈 테스트 커버리지 약 30%

Wolf RPG 모듈(15개 파일), IPC 핸들러(4개), apply.ts, projectConvert.ts 등 핵심 모듈에 테스트 없음. 검증(verify.ts)과 번역(translator.ts) 모듈은 37개 이상으로 충실.

test/unit/ — 8개 파일로 전체 55개 TS 모듈 중 ~15개만 커버
의도적 트레이드오프
Windows 전용 빌드 · 경로 규칙

electron-builder가 win 타겟만 설정하고, CI도 windows-latest에서만 실행. 대상 사용자가 Windows 게이머이므로 의도적이나, 크로스 플랫폼 확장성이 없다.

3. 기술 · 보안

비의도적 결함
electron-store 암호화 키 하드코딩

getEncryptionKey()가 'tsukuru-extractor-store-key'를 평문으로 파일에 기록하고 그대로 사용. LLM API 키가 이 키로 암호화되므로 실질적 보호가 없다.

src/ipc/shared.ts:17
강점
Electron 보안 모델 충실 적용

contextIsolation: true, nodeIntegration: false, IPC 채널 화이트리스트(28 send + 20 receive), nodeFs 경로 화이트리스트. sandbox: false는 Electron 40에서 preload + contextIsolation 조합 시 필수 제약으로, 문서화된 의도적 트레이드오프.

src/preload.ts, src/ipc/windowManager.ts
비의도적 결함
의존성 보안 감사 부재

npm audit나 Dependabot 미설정. 18개 프로덕션 의존성 중 jsdom(28.1)은 서버 컨텍스트에서 XSS 벡터 가능성 있으나, 데스크톱 앱에서의 실질적 위험은 낮음.

4. 코드 품질 · 구조적 리스크

비의도적 결함
비일관적 에러 처리 — IPC 보호 vs 파일 I/O 미보호

IPC 핸들러들은 try/catch로 감싸져 있으나, apply.ts, fileCrypto.ts, Wolf 모듈의 readFileSync/writeFileSync 20개 이상이 에러 처리 없이 호출. 손상된 게임 파일 입력 시 프로세스 크래시 위험.

src/js/rpgmv/apply.ts, src/js/wolf/applyWolf.ts, src/js/rpgmv/fileCrypto.ts
비의도적 결함
128개 ESLint 경고 (주로 any 타입)

TypeScript strict 모드를 통과하나, @typescript-eslint/no-explicit-any 경고 128개 잔존. 타입 안전성의 마지막 마일이 미완성.

강점
JSON 검증 모듈(verify.ts)의 설계 품질

의미론적 앵커, 이벤트 구조, 제어문자, 커스텀 메타데이터 4계층 검증 + LLM 자동 수정까지 구현. 37개 테스트로 커버. 프로젝트의 핵심 가치를 가장 잘 구현한 모듈.

src/js/rpgmv/verify.ts (571 LOC), test/unit/verify.test.ts (37 tests)
D. 액션 플랜
Quick Win
암호화 키를 OS 키체인으로 이전

Windows DPAPI 또는 keytar 패키지로 electron-store 암호화 키를 시스템 수준에서 관리. 현재 평문 키 파일 → 리버스 엔지니어링으로 LLM API 키 유출 가능.

Hardening
파일 I/O 에러 처리 일관화

readJsonFile(), writeJsonFile() 같은 공통 유틸리티를 만들어 BOM 제거 + JSON 파싱 + try/catch를 일원화. 현재 25개 이상의 수동 패턴을 통합.

Strategic
이중 파일 규칙 제거 — tsc 빌드 스텝 도입

Main 프로세스용 tsconfig.main.json으로 outDir 빌드를 자동화하고, 커밋된 .js 동반 파일 44개(5,260 LOC)를 제거. 동기화 실수를 원천 차단.

Hardening
테스트 커버리지 확대 — Wolf · apply · IPC 모듈

현재 ~30% 모듈 커버리지를 60% 이상으로 확대. 우선순위: apply.ts(추출-적용 파이프라인 핵심), Wolf RPG extractor.ts, IPC 핸들러 통합 테스트.

Quick Win
ESLint 경고 128개 해소 + Dependabot 설정

남은 any 타입을 구체화하고, .github/dependabot.yml을 추가하여 의존성 보안 업데이트 자동화.

Strategic
globalThis 전역 상태를 DI 컨테이너로 리팩터링

16개 전역 변수를 AppContext 클래스에 캡슐화하여 모듈 간 명시적 의존성 주입. 테스트 격리와 코드 이해도를 동시에 개선.

Quick Win
중복된 rmBom 함수 제거

src/utils.ts의 rmBom은 테스트만 존재하고 프로덕션 코드에서 미사용. projectTools.ts 버전과 통합하거나, 파일 I/O 유틸리티 통합 시 함께 정리.

일본어 RPG 게임 번역을 자동화하는 Windows 데스크톱 앱. 텍스트 추출 → LLM 번역 → JSON 재조립 파이프라인을 Electron + Vue 3 + TypeScript로 구현.

최신 스택 위에서 도메인 특화 기능(줄밀림 검증, LLM 자동 수정)을 잘 구현했다. JSON 검증 모듈은 37개 테스트와 함께 프로덕션 수준의 완성도를 보여주며, CI/CD 파이프라인과 Electron 보안 모델도 충실하다. 그러나 프로젝트의 골격에는 빠른 반복 개발의 흔적이 곳곳에 남아있다 — 44개 JS 동반 파일 수동 동기화는 언제든 동기화 사고를 일으킬 수 있는 시한폭탄이고, 16개 전역 변수에 의존하는 상태 관리는 모듈 경계를 무의미하게 만든다. 파일 I/O의 에러 처리가 IPC 계층에서는 꼼꼼하지만 비즈니스 로직 계층에서는 빠져 있어, 손상된 게임 파일 하나로 전체 앱이 크래시할 수 있다.

핵심 기능은 올바르게 동작하고 사용자 가치를 제공하지만, 구조적 부채를 해소하지 않으면 기능 추가마다 버그 발생 확률이 누적된다. 정식 배포에는 경화(hardening) 단계가 필요하다.

⚖️ 프로덕션 준비도 판정: 조건부 통과 — 암호화 키 보안과 파일 I/O 에러 처리를 해결한 후 배포 가능. 이중 파일 규칙과 전역 상태는 중기 개선 과제.
