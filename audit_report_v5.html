C. 핵심 발견 사항

1. 지속 가능성 (6.5/10)

✅ 체계적 모듈 분리
main.ts 40줄로 축소, 6개 IPC 핸들러 모듈 분리, 추출 파이프라인 4파일 분할. compareUtils.ts에 순수 로직 분리는 테스트 용이성을 크게 높임.
⚠️ 이중 파일 컨벤션 (.ts + .js)
메인 프로세스 .ts 파일마다 커밋된 .js 컴패니언 46개를 수동 동기화해야 함. vitest.config.ts에서 resolve.extensions 순서를 바꿔야 .ts가 우선 로드되는 등, 새 개발자가 빠지기 쉬운 함정. 의도적 트레이드오프지만 onboarding 비용이 높다.
⚠️ 전역 가변 상태 의존
appContext.ts의 appCtx 싱글톤에 mainWindow, settings, gb, WolfCache 등 앱 전체 상태가 집중. 크기 제한 없는 gb·WolfCache는 대형 게임에서 메모리 문제 가능. globalThis → appCtx 마이그레이션은 긍정적이나, 여전히 전역 뮤터블 객체.

2. 엔지니어링 재현성 (7.5/10)

✅ 완전한 CI 파이프라인 + 190개 테스트
.github/workflows/ci.yml: tsc → lint → test:coverage → electron-builder 검증. 10개 테스트 파일에서 190개 테스트 모두 통과. package-lock.json 커밋, npm ci 사용.
✅ README 한국어 문서화
readme.md에 설치·실행·빌드·테스트 명령어와 프로젝트 구조가 명시됨. CHANGELOG.md에 3.0.0 변경 내역 정리.
⚠ 이중 파일 빌드 규칙 미문서화
.ts/.js 이중 파일 컨벤션, git add -f 필요성, vitest resolve.extensions 설정의 이유가 readme.md에 없음. .github/copilot-instructions.md에만 기록되어 있어 새 개발자가 인지하기 어렵다.

4. 코드 품질 및 구조 위험 (6.0/10)

✅ 핵심 파이프라인 테스트 커버리지
추출-적용 라운드트립(extract-apply.test.ts), 번역 블록 검증(translator.test.ts 37개), JSON 구조 검증(verify.test.ts 37개), 비교 유틸(compareUtils.test.ts 25개)에 대한 충실한 테스트.
🔴 Wolf 바이너리 파서 무한 루프 위험
wolf/parser/io.ts:26-31 — findByteArray()가 바이트 시퀀스를 찾지 못하면 while(true)로 무한 루프. patternBased.ts:46의 이벤트 파싱 루프도 동일한 위험. 손상된 Wolf 파일 입력 시 앱이 100% CPU를 소비하며 행(hang).
⚠️ apply.ts 경로 오타 — 잠재적 데이터 손실
apply.ts:31-35 — .Completed 존재 시 /Completed(점 없음)를 삭제 시도. 서로 다른 디렉토리를 참조하므로 .Completed가 이미 존재하면 기존 Completed 폴더가 삭제되지만 새 .Completed는 생성되지 않아 완료 상태가 유실될 수 있음.
⚠ 매직 넘버·한국어 하드코딩 산재
이벤트 코드 401/108/355/655, 자동줄바꿈 임계값 80/60, 정렬 모드 문자열 등이 상수 정의 없이 사용. UI 메시지는 전부 한국어 리터럴로, 국제화 가능성 없음. 단일 사용자 한국어 도구라는 맥락에서 의도적이나 유지보수 시 파편화 위험.

D. 조치 계획

1
라이선스 불일치 해결 Quick Win
package.json의 "license": "MIT"와 LICENSE 파일(GPL-3.0)의 모순 해소. 배포 전 반드시 하나로 통일.
2
Wolf 파서 무한 루프 방지 Hardening
wolf/parser/io.ts의 findByteArray()와 patternBased.ts의 이벤트 파싱 루프에 버퍼 길이 기반 bounds check 추가. 최대 반복 횟수 또는 포인터 범위 체크.
3
apply.ts 경로 오타 수정 Quick Win
apply.ts:32의 '/Completed'를 '/.Completed'로 수정하여 디렉토리 이름 일관성 확보.
4
WolfDec SHA-256 검증 활성화 Hardening
extentions.ts:10의 빈 해시를 실제 SHA-256 값으로 채우거나, 해시 검증을 사용자에게 경고하는 UI 피드백 추가.
5
이중 파일 컨벤션 문서화 또는 자동화 Strategic
readme.md에 .ts/.js 이중 파일 규칙 명시하고, prestart/prebuild의 tsc 스텝이 이를 자동 동기화함을 설명. 장기적으로는 preload.ts를 번들러(esbuild)로 처리하여 .js 컴패니언 제거 검토.
6
vitest.config.js 중복 파일 제거 Quick Win
vitest.config.js는 vitest.config.ts의 tsc 컴파일 산출물. .gitignore에 추가하거나 삭제하여 혼란 방지.
7
매직 넘버 상수화 Strategic
parser.ts·apply.ts의 이벤트 코드(401, 108, 355 등), 자동줄바꿈 임계값(80, 60)을 datas.ts에 명명된 상수로 정의.
