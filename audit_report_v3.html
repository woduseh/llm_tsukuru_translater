A. 프로젝트 개요 & 기술 스택

C. 주요 발견 사항

1. 지속가능성 (7.0/10)

✅ 모듈 분리 완수: main.ts 750→37줄, extract.ts 577줄→4파일 분할(extract/parser.ts, formatter.ts, io.ts, index.ts), renderer.ts 618줄→6파일 분할. IPC 핸들러 6개 모듈로 체계화. 의도적 구조 개선 — 각 모듈이 단일 책임을 가짐.
✅ 타입 체계 구축: src/types/settings.ts에 AppSettings(27필드), src/js/rpgmv/types.ts에 RPG Maker 타입 20+ 인터페이스 정의. any 113→20건(-82%), ESLint 경고 248→131건(-47%). 명시적 타입이 코드 이해도를 높임.
⚠️ 구조적 로깅 부재: 전체 코드베이스가 console.log/warn/error에 의존. main.ts:32의 uncaughtException 핸들러가 console.log만 호출 — 프로덕션 크래시 디버깅 불가. Winston/Pino 같은 구조적 로깅 없음. 의도적 간소화가 아닌 누락 — 크래시 로그 파일(debug_crash.log) 존재가 필요성을 반증.

2. 재현성 (7.5/10)

✅ 테스트 인프라 견고: 136개 테스트(8 스위트) — 추출↔적용 라운드트립(21), 번역기 블록 로직(37), 암복호화(11), AppContext 동기화(21), 유틸리티(17), JSON 무결성(18). vitest run 332ms 완주. 엣지 케이스(빈 맵, null 이벤트, 멀티라인, BOM) 포함.
✅ CI 파이프라인: .github/workflows/ci.yml — Windows runner, Node 20, npm ci → tsc → lint → test. package-lock.json 커밋, 의존성 핀 완료.
⚠️ README 문서 부족: readme.md는 4줄 분량으로 설치·설정·개발 가이드 없음. 위키 참조만 존재하나 로컬 문서 미포함. .env.example, CHANGELOG 부재. 오픈소스 데스크톱 도구로서 최소한의 설치 가이드 필요.

3. 보안 (6.5/10)

✅ Electron 보안 모델 적용: contextIsolation: true, preload 스크립트의 채널 화이트리스트(21 send, 15 receive), API 키 x-goog-api-key 헤더 전송, electron-store 암호화. innerHTML 0건 달성(15건→0건). nodeFs에 경로 화이트리스트 도입(preload.ts:10-14).
⚠️ 경로 화이트리스트 기본값 허용: preload.ts:11 — if (allowedBasePaths.length === 0) return true. 앱 시작 직후 또는 폴더 미선택 상태에서 모든 경로 접근 가능. 기능 호환성을 위한 트레이드오프이나, 기본 거부가 안전한 디폴트.
⚠️ 하드코딩된 암호화 키: src/ipc/shared.ts:6 — encryptionKey: 'tsukuru-extractor-store-key'가 소스에 노출. 또한 decrypter.ts:11에서 exec()를 사용한 외부 바이너리 실행 — execFile()이 안전. exec()는 shell injection 가능성이 있으나, 경로가 내부 생성이므로 실질 위험은 낮음.

4. 코드 품질 (7.0/10)

✅ globalThis.mwindow 완전 마이그레이션: 대상 모듈(apply, translator, fileCrypto, projectConvert)에서 47건 제거 완료. Tools.send() 추상화 일관 적용. 잔여 6건은 인프라 모듈(windowManager, settingsHandler)에서의 정당한 사용.
✅ 타입 안전성 대폭 향상: strict 모드 하에서 any 20건만 잔존(대부분 IPC 콜백의 ...args: any[]와 서드파티 Swal 타입). src/js/rpgmv/types.ts(253줄)에 RPG Maker 전체 데이터 구조 타입화.
⚠️ unknown 캐스팅 검증 부재: 렌더러 전반에서 IPC 수신 데이터를 (tt as Record<string, unknown>)로 캐스팅하나 런타임 검증 없음. 형상 불일치 시 런타임 크래시 가능. apply.ts:97에 빈 catch 블록 1건 잔존. 빈 catch는 JSON.parse 실패 시 기존 데이터 유지 의도로 보이나 명시적 주석 없음.

이 프로젝트는 적극적인 구조 개선을 거쳐 프로토타입 수준에서 생산 임계점까지 올라왔다. 750줄짜리 진입점을 37줄로 줄이고, 500줄 이상 파일 두 개를 분할하고, 전역 상태 47건을 추상화 레이어로 교체한 것은 의미 있는 엔지니어링 투자다. 136개 테스트가 핵심 파이프라인(추출→적용 라운드트립, 암복호화, 블록 분할)을 실질적으로 커버하고, 332ms에 전량 통과하는 것은 회귀 방지 기반이 잘 잡혀 있음을 보여준다. 그러나 보안 모델은 "구현은 했으나 디폴트가 안전하지 않은" 상태(빈 화이트리스트=전체 허용, 하드코딩 키)이고, 프로덕션 디버깅을 위한 구조적 로깅이 전무하며, IPC 데이터의 런타임 형상 검증 없이 타입 단언에만 의존하는 점은 B등급 상한을 가로막는 실질적 약점이다. 문서화가 위키 외부 의존에 머물러 있어, 새 기여자의 온보딩은 코드 직접 읽기에 의존할 수밖에 없다.
